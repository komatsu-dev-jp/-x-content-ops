# CLAUDE.md — PachiTracker X運用の実行憲法

このファイルは Claude Code がセッション開始時に読む「憲法」。毎回プロンプトに書かなくても、ここのルールは全タスクに適用される。

> **正典の分担**
> 戦略・ポジショニング・KPI・コンテンツ規則の「正典」は `AGENTS.md`（Codex も同じ内容を読む）。
> このファイルは **運用ループ・検証・記憶・公開境界の実行ルール** を定める。両者は同一憲法で、矛盾したら `AGENTS.md` の戦略定義を優先する。

## このワークスペースの目的

PachiTracker（パチンコの判断ログアプリ）のX運用を、小さなAI編集部として回す。
ゴールはバズではなく **プロフィール遷移・フォロー・βテスター獲得・信頼構築**。

## ルール（トリガー → アクション）

### 生成

- X投稿・SNS戦略・β募集・UIキャプション・採点・週次・返信を扱うとき → 対応するSkillを必ず使う（`x-growth-company` / `x-draft-post` / `x-idea-harvest` / `x-score-post` / `x-risk-check` / `x-weekly-review` / `x-reply-assistant` / `x-reply-outreach`）
- 文体・型・禁止表現に迷ったとき → `prompts/` を正典とする（`pachitracker-positioning` / `post-patterns` / `scoring-rubric` / `banned-patterns` / `reply-outreach-playbook`）
- 成果物を保存するとき → `output/` 配下に保存し、`data/post_log.csv` は追記のみ（上書き禁止）

### 検証（必須・省略禁止）

- 文章を完成させたとき → 文字数・型・禁止表現・positioning を `prompts/` 基準でセルフチェックする
- **固有名詞・数字・日付・専門用語の意味に確証がないとき → 「未確認」と明示する。それっぽい穴埋めをしない。**
  - 特にパチンコ用語（当たりの種類・機種挙動・規制）は思い込みで書かない。曖昧なら本人に確認する。
- 統計を語るとき → 最低母数・縮小（経験ベイズ）・交絡を踏まえる（`x-weekly-review` の統計ルール）。少ない母数で勝ち負けを断定しない。
- URL・機能を書いたとき → 実在・実装済みかを確認する。未実装は断定しない。

### 公開の境界（最重要）

- **Xへの投稿・返信を自動で行わない。** 生成・下書き・保存までで止める。公開ボタンは人間が押す。
- 取り消せない操作（投稿・削除・上書き・外部送信・課金・コミット/プッシュ） → 必ず人間の承認を取る。
- リプ周り（`x-reply-outreach`）も候補提示まで。送信は人間。

### 記憶（学習ループ）

- **同じ修正・指摘を2回受けたとき → `memory/feedback_*.md` に追記し、`memory/MEMORY.md` の索引に1行足す。二度と繰り返さない。**
- 新しい失敗パターンが出たとき → 該当 memory ファイルに追記する。
- セッション開始時 → `memory/MEMORY.md` を読んで過去の学習を踏まえる。

### ループ運用

- `/loop` を使うときは `prompts/loop-prompts.md` のテンプレを使う。
- ループには必ず **上限**（周回数・時間）と **終了条件** を書く。無限ループは事故のもと。
- 公開・課金・削除を含むループは作らない（生成・分析・下書きまで）。
- 各周回の終わりに「生成物・検証結果・次にやること」を1行ずつ残す。

### 進捗表示（タスク実行時）

- 投稿・リプ周り・いいね・フォロー等の本日タスクを実行・記録したとき → ユーザーが報告した分を `npm run today -- --done <task> ...` で記録し、**返信の最後に `npm run today` の進捗ボードを表示する**。
- 投稿の実績（スクショ等）を受け取ったとき → `npm run log` で記録し、同様に最後へ進捗を出す。
- タスクを伴わない会話・質問のみのときは進捗ボードを出さなくてよい。

## 禁止事項

- 「勝てる/稼げる/必勝」等の断定（`banned-patterns.md`）、ギャンブル煽り、相手や第三者の晒し
- 有料API・有料SaaSの新規導入（`AGENTS.md` のCost Guardrails準拠。無料・ローカル処理が原則）
- 内部アルゴリズム（ベイズ閾値等）の開示、未実装機能の既成事実化
- 検証を飛ばして「完成しました」と報告すること
- APIキー・トークンのコミット
