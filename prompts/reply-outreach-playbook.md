# リプ周り Playbook（他人の投稿への返信で「返信が返ってくる」設計）

このファイルは `x-reply-outreach` Skill が読む。
目的は **コールド返信（他人の投稿へのリプライ）で、相手とそのフォロワーをPachiTrackerに呼び込むこと**。
自分の投稿への返信は `x-reply-assistant` を使う（別物）。

---

## 大前提（これを外すと逆効果）

- **自動化しない。** 候補を出すだけ。投稿は人間が選んで手で送る。
- **関連のない投稿に送らない。** パチンコの台選び・続行・撤退・収支・メンタル・記録、データで考える文化——この文脈に重なる投稿だけ。
- **コールド返信にリンクを貼らない。** リーチが落ち、スパム臭が出て、ブロック/通報リスクが上がる。PachiTrackerへの導線は固定ポストとプロフィールに任せる。
- **売り込まない。** 機能・アプリ名・β募集をコールド返信に書かない。思想が「にじむ」程度で止める。相手が興味を示してから初めて出す。
- **1日の上限を守る。** 質の高い返信を1日3〜7件。連投は同一スレ巡回botに見える。
- **景表法・規約・煽り禁止。** 「勝てる」「稼げる」断定、当確煽り、相手・他者の晒し、内部情報は出さない（`x-risk-check` を必ず通す）。

---

## 「返信が返ってくる」返信の3原則

1. **価値を1つ足す。** ただの「分かります」「最高です」は捨てる。新情報・具体例・別角度・一次体験のどれかを必ず1つ入れる。
2. **答えやすい質問を1つだけ。** 広すぎる質問（「どう思いますか？」）は返ってこない。**1行で答えられる狭い質問**にする（例:「一番ブレるのは台選び・続行・撤退のどこですか？」）。
3. **短く、相手の言葉を拾う。** 1〜2文。相手が使った単語を1つミラーリングすると「読んだ感」が出て返信率が上がる。

---

## 返信の型（archetype）

| 型 | 使う場面 | 狙い | 例 |
|---|---|---|---|
| 一次情報追加 | 相手が主張・感想 | 自分の具体体験/数字で会話の燃料を足す | 「分かります。自分も昨日、回ってた記憶につられて1000円多く突っ込みました。」 |
| 狭い質問 | 相手が問題提起 | 1行で答えられる問いで返信を引き出す | 「ちなみに撤退が遅れるのは、投資額が理由ですか？それとも“次で来る”感覚ですか？」 |
| 同意＋一歩 | 相手の意見に賛成 | 肯定してから半歩深め、考えを引き出す | 「ほんとそれです。しかもその“理由”、後から見ると感情の後付けだったりしますよね。」 |
| あるある言語化 | 共感狙い | 相手の痛みを一段クッキリ言語化して刺す | 「『勝った日の記憶』だけ残るの、ほんと厄介ですよね。」 |
| やさしい視点転換 | 断定・極論に | 否定せず別角度を1つ置く（マウント厳禁） | 「なるほど。逆に“続けた日”の理由って記録に残してます？そこ見ると景色変わること多くて。」 |

迷ったら **一次情報追加＋狭い質問** の合わせ技が最も返信されやすい。

---

## やってはいけない（アンチパターン）

- 中身のない称賛（「最高です」「神」「勉強になります」だけ）
- 絵文字だけ・スタンプ的リプ
- 大喜利・すべりリスクのあるネタ
- 説教・上から目線・マウント（「それは○○が悪い」）
- いきなり宣伝・アプリ名・リンク
- 長文（3文以上）
- 相手の言葉を一切拾わないテンプレ感
- 否定から入る／論破しにいく

---

## ターゲット選定（誰に返すか）

**向く投稿:**
- パチンコの台選び・続行・撤退・収支・メンタル・記録の話
- 「なぜ負けたか」を言語化しようとしている投稿
- データ/期待値/立ち回りを“感覚でなく”考えたい層
- 中規模でリプに反応してくれるアカウント（巨大bot垢よりエンゲージが返る）

**避ける投稿:**
- 機種スペック速報bot・アフィ・当確煽り垢・釣り垢
- 炎上・ネガティブ絡みのスレ
- 政治/誹謗中傷など文脈外

---

## タイミング

- 投稿から **数十分以内** の、伸び始めている投稿に早く付けると上位表示されやすい。
- 既に何百もリプが付いた投稿は埋もれる。リプが少なく勢いのある投稿が狙い目。

---

## KPI（コールド返信の効果測定）

主軸は **返信が返ってきたか → プロフィール遷移 → フォロー**。いいねは参考程度。
記録する場合は `data/reply_outreach_log.csv`（任意）に
`date, target_url, tier, archetype, got_reply(0/1), profile_visit(0/1), follow(0/1), note` を残し、
週次でどの型が返信を取れているかを見る。

## ターゲットの tier 比率（Notion「対象比率」基準）

リプ周りの対象は、フォロワー規模で次の比率を目安に配分する:

| tier | フォロワー規模 | 目標比率 | 狙い |
|---|---|---|---|
| A | 500〜5,000（濃い収支・期待値垢） | 50% | 読まれやすく、相互化しやすい |
| B | 5,000〜50,000（中堅アカウント） | 30% | 露出と信頼獲得 |
| C | 10万超（万人アカウント） | 20% | 当たれば大きいが埋もれやすい |

リプ案を出すときはターゲットの tier を分かる範囲で添える。フォロワー数が画面から読めない場合は「不明」と明示し、推測で断定しない。

50,000〜100,000は表の隙間のためBに含める。500未満の小型アカウントは暫定でAに含める（A本来の下限より小さい点は記録時に明示する）。

### アカウント登録（2回目以降は自動でtierが付く）

`data/account_tiers.csv` にハンドルとフォロワー数を登録しておくと、以降そのアカウントへの `npm run today -- --done reply` で tier を省略しても自動で引かれる。

```
# フォロワー数が分かったら登録（tierは自動計算）
npm run reply-log -- --set-account handle=matutakenet followers=14113
```

### 記録コマンド

```
# 送信直後（1コマンドで日次カウント＋tier比率の両方に記録される）
npm run today -- --done reply target_url=<URL> tier=A/B/C archetype=<型> note=<メモ>
# tier省略時はaccount_tiers.csvに登録があれば自動参照、なければ未設定のまま

# 後日、反応が分かったら
npm run reply-log -- --update target_url=<URL> got_reply=1 profile_visit=1 follow=0

# 比率の実績を確認（累計 or 直近N日）
npm run reply-mix
npm run reply-mix -- --days 7
```
