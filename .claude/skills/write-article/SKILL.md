# write-article スキル

AIツール・サービスの最新アップデートに関する記事を、超初心者でもわかるように執筆するスキル。
毎回同じ品質の記事を再現性高く生成するためのワークフローを定義している。

## 概要フロー

1. **STEP 1: 入力受け取り** — テーマ・読者・参考記事を確認
2. **STEP 2: リサーチ** — 公式情報源から最新情報を収集・照合
3. **STEP 3: スタイル確認** — 既存記事またはstyle-guide.mdからスタイルを把握
4. **STEP 4: 記事執筆** — 15,000文字以上、初心者向け、実務重視
5. **STEP 5: スクリーンショット撮影** — Claude in Chrome で関連ページを撮影
6. **STEP 6: 3人採点パネル** — 95点以上で通過、未満は修正して再採点
7. **STEP 7: 保存** — output/posts/ にファイル保存

---

## STEP 1: 入力受け取り

ユーザーから以下の情報を確認する（明示されていない場合はデフォルトを使用）:

| 項目 | 説明 | デフォルト |
|------|------|-----------|
| `テーマ` | 記事の主題（例: "2026年4月Claude Codeアップデート"） | 必須 |
| `ターゲット読者` | 誰向けか | 超初心者・非エンジニア |
| `参考にする既存記事` | スタイル参考用（省略可） | output/posts/ 内の最新記事 |
| `出力先` | 保存先ディレクトリ | `output/posts/` |

出力先ディレクトリが存在しない場合は作成する。

---

## STEP 2: リサーチ

WebSearch と WebFetch を使って情報を収集する。

**収集する情報:**
- 公式 CHANGELOG・リリースノート（GitHub等）
- バージョン番号・リリース日付
- 機能の正式名称と説明
- 対象ユーザー・プラン情報

**照合の原則:**
- 一次情報（公式ドキュメント・GitHub）を最優先とする
- 数値（バージョン番号・パーセンテージ等）は特に慎重に確認
- 「〜らしい」「〜と思われる」などの推測表現は使わない。データがない場合は「確認が必要」と明示する

---

## STEP 3: スタイル確認

記事を書く前に、`output/posts/` ディレクトリ内の既存記事を1つ読んでスタイルを把握する。
既存記事がない場合は `references/style-guide.md` を参照すること。

**必ず守るスタイルルール（詳細は references/style-guide.md 参照）:**

- **書き出し**: 個人的なエピソード・情景描写で始める
- **章タイトル**: `## ─── 第X章：タイトル ───` 形式で統一
- **文体**: 口語・体験談ベース。AI臭（「〜が可能になります」等）を排除
- **専門用語**: 必ず括弧で説明を入れる
- **比較表**: 各機能の説明に比較表または使用例テーブルを入れる
- **実用性**: 各機能に「仕事でこう使える」具体例を必ず1つ以上

---

## STEP 4: 記事執筆

以下の構成で記事を書く。文字数は15,000文字以上（理想は20,000文字以上）。

### 推奨構成

```
# 記事タイトル（具体的・数字入り）

**公開日: YYYY年M月D日**

## ─── はじめに ───
（個人的なエピソードで引き込む。2〜3段落）

## ─── 第1章：そもそも「〇〇」って何？（完全初心者向け） ───
- 一言で何か
- 他のツールとの違い（比較表）
- 使うと何が変わるか（before/after）

## ─── 第2章：今月のアップデート全体像 ───
- 概要テーブル（機能名・バージョン・一言まとめ・仕事への影響★）

## ─── 第3章〜第N章：主要機能の詳細 ───
（各機能ごとに: 何なのか → どう使うか → 仕事での具体例 → スクリーンショット）

## ─── 第N+1章：地味だけど助かる細かい改善 ───

## ─── 第N+2章：今日から始める方法 ───
- インストール手順・プランと料金表

## ─── 第N+3章：初心者向け実践ガイド ───
- よくある不安への回答・最初の5日間プラン（表形式）

## ─── まとめ ───
- 設定不要ですぐ使えるもの
- 明日試してほしいもの
```

### 執筆後の自己チェック（必須）

- [ ] 文字数15,000以上
- [ ] はじめにがエピソードで始まっている
- [ ] 章タイトルが統一形式（`## ─── 第X章：～ ───`）
- [ ] 専門用語に括弧説明あり
- [ ] 各機能に「仕事での使い方」が最低1ケース
- [ ] AI臭のある表現がない
- [ ] 架空の数値サンプルに「概算例」注釈あり
- [ ] PDFの操作例は「テキスト系PDF（文字検索できるPDF）」と条件明記

## STEP 5: スクリーンショット撮影

スクリーンショットは `output/posts/screenshots/` ディレクトリに保存する。

### 撮影手順

1. `mcp__Claude_in_Chrome__tabs_context_mcp` で `createIfEmpty: true` を指定してタブIDを取得
2. `mcp__Claude_in_Chrome__navigate` で目的のページに移動（2〜3秒waitを入れる）
3. `mcp__Claude_in_Chrome__computer` の `screenshot` アクションで内容を確認
4. PowerShell で画面をキャプチャして正しいファイルパスに保存:

```powershell
Add-Type -AssemblyName System.Windows.Forms
Add-Type -AssemblyName System.Drawing
$screen = [System.Windows.Forms.Screen]::PrimaryScreen
$bitmap = New-Object System.Drawing.Bitmap($screen.Bounds.Width, $screen.Bounds.Height)
$graphics = [System.Drawing.Graphics]::FromImage($bitmap)
$graphics.CopyFromScreen($screen.Bounds.Location, [System.Drawing.Point]::Empty, $screen.Bounds.Size)
$bitmap.Save("C:\FULL\PATH\TO\screenshots\NN_description.png", [System.Drawing.Imaging.ImageFormat]::Png)
$graphics.Dispose(); $bitmap.Dispose()
```

### ドメイン制限

| ドメイン | 状況 | 対処 |
|---------|------|------|
| anthropic.com | スクリーンショット可 | 積極的に使う |
| claude.com | スクリーンショット不可 | anthropic.com の関連ページで代替 |
| github.com | ナビゲーション不可 | 他サービスで代替 |

**キャプションは実際の画像内容と必ず一致させること。**
anthropic.com の画像を使った場合は「Anthropic公式サイトより」と明記する。

## STEP 6: 3人採点パネル

詳細な採点基準は `references/review-criteria.md` を参照。

### 採点ペルソナ（各50点満点）

**審査員A：情報正確性**
- バージョン番号・機能名・対象ユーザーの正確性
- 架空数値への注釈・推測表現の排除

**審査員B：文章品質（辛口）**
- AI臭の残存チェック・構成の一貫性
- 読者目線での実用性・誤解を招く表現
- スクリーンショットキャプションと画像の整合性

**ファシリテーター：総合調整**
- 2人の意見を統合して最終スコア算出（100点満点）
- 修正指示を【必須】【推奨】の2段階で整理

### 採点ルール

```
95点以上 → 通過・STEP 7 へ
95点未満 → 【必須】修正を実施 → 再採点（最大3回）
```

## STEP 7: 保存

### ファイル保存

```
記事: output/posts/YYYY-MM-DD_article-slug.md
スクリーンショット: output/posts/screenshots/NN_description.png
```

### DAILY.md への記録（ファイルが存在する場合のみ）

```markdown
### 記事執筆完了
- タイトル: [記事タイトル]
- ファイル: output/posts/YYYY-MM-DD_article-slug.md
- 文字数: XX,XXX文字
- 採点結果: XX/100点
- 完了時刻: HH:MM
```
