# 記事スタイルガイド

このガイドは「ナレッジ系記事」のスタイルを定義します。
記事を書く際は必ずこのガイドに従ってください。

---

## 基本方針

- **読者**: 超初心者〜中級者（ITリテラシーは最低限）
- **トーン**: フレンドリー、でも信頼できる先輩の話し方
- **文体**: 常体（だ・である調）は避け、敬体（です・ます調）を基本に
- **目標文字数**: 10,000字以上（理想は15,000字以上）

---

## 冒頭（書き出し）パターン

必ず以下のパターンのいずれかで書き出す。AIっぽい「本記事では〜」は絶対禁止。

### パターンA：エピソード型（推奨）
実際の体験談や具体的シーンから入る。
```
例：
「先週、取引先との打ち合わせ資料を30分で仕上げる羽目になった。
去年の自分なら絶望していたと思う。でも今は違う。」
```

### パターンB：問いかけ型
読者に問いかけてから入る。
```
例：
「あなたは1日何時間、"探す作業"に使っていますか？
実は平均的なビジネスパーソンは、1日2.5時間を情報検索に費やしているらしい。」
```

### パターンC：数字インパクト型
驚きの数字や事実から入る。
```
例：
「3秒。それが、AIが議事録を要約するのにかかる時間です。」
```

## 章構成のルール

### 章タイトルのフォーマット
```
## 第1章：〇〇〇〇〇
## 第2章：〇〇〇〇〇
```
（必ず「第N章：」から始める）

### 注目機能の見出しフォーマット
複数の注目機能を紹介する場合は統一した書き方にする：
```
### 注目① 機能名
### 注目② 機能名
### 注目③ 機能名
```
（「激推し①」「イチオシ①」などのバラバラな表現は禁止）

---

## 表・比較のフォーマット

機能一覧や比較は必ず表で示す。表には「何に使えるか」列を含める。

```markdown
| 機能名 | できること | 仕事での活用例 |
|--------|-----------|---------------|
| A機能  | 〇〇できる | 〇〇に使える   |
```

---

## AI臭チェックリスト（禁止表現）

以下の表現が含まれていたら書き直す：

- 「本記事では〜」「本稿では〜」
- 「〜について解説します」（単独で使う場合）
- 「〜ということになります」
- 「〜という観点から」
- 「非常に重要です」「極めて重要です」
- 「〜を踏まえると」
- 「まとめると以下の通りです」（単独で使う場合）
- 「〜に他なりません」
- 「ぜひ活用してみてください」（口癖になっていたら削減）
- 「〜することができます」→「〜できます」に言い換え
- 「〜について触れておきます」

---

## 仕事での活用例の書き方

機能紹介には必ず「仕事でこんな風に使える」具体例を入れる。

**NGパターン（抽象的）:**
```
> この機能を使えば作業効率が上がります。
```

**OKパターン（具体的なシーン）:**
```
> 例えば、会議後に「さっきの議事録を整理して、
> 担当者ごとにアクションアイテムを箇条書きにして」と
> 頼むだけで3分以内に完成する。
```

## スクリーンショットのキャプション

スクリーンショットは以下の形式で入れる：

```markdown
![説明](screenshots/ファイル名.png)
*▲ キャプション文（何が写っているか＋何のために見せるか）*
```

キャプションの注意：
- 実際のスクリーンショットの内容を正確に説明する
- 「公式サイト」「Anthropicのサイト」など場所も明記する

---

## まとめセクション

最後は「まとめ」セクションを設け、以下を含める：
1. 今回の内容を3〜5行でサマリー
2. 「次のアクション」として読者がすぐできることを1〜3個提示
3. 感情的なクロージング（「一緒に試してみましょう」系）
